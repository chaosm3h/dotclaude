# テスト駆動開発

## 理念

**基本原則**：テストは実装の詳細ではなく、公開インターフェースを通じて動作を検証すべきである。コードは完全に変更されても、テストは変更されるべきではない。

**良いテスト**は統合テストのスタイルをとる：公開APIを通じて実際のコードパスを実行する。それらはシステムが「何」を行うかを記述し、「どのように」行うかは記述しない。優れたテストは仕様書のように読めるものです。「ユーザーは有効なカートでチェックアウトできる」という記述は、どのような機能が存在するかを正確に伝えます。こうしたテストは内部構造を気にしないため、リファクタリング後も存続します。

**悪いテスト**は実装に依存しています。内部の連携先をモックしたり、プライベートメソッドをテストしたり、外部手段（インターフェースを使用せずにデータベースを直接クエリするなど）を通じて検証したりします。警告サイン：リファクタリングを行った際にテストが失敗するが、動作自体は変わっていない場合です。内部関数の名前を変更しただけでテストが失敗するなら、そのテストは動作ではなく実装をテストしていたことになります。

例については [tests.md](tests.md) を、モックのガイドラインについては [mocking.md](mocking.md) を参照してください。

## アンチパターン：水平スライス

**まずすべてのテストを書き、その後で実装をすべて書くようなことはしないでください。** これは「水平スライス」と呼ばれるもので、REDを「すべてのテストを書くこと」、GREENを「すべてのコードを書くこと」と捉える考え方です。

これにより、**質の低いテスト**が生成されます：

- 一括で書かれたテストは、"実際の動作"ではなく、"想像上の動作"をテストしてしまう
- "ユーザー向けの動作"ではなく、"物事の構造（データ構造や関数のシグネチャ）"をテストすることになってしまう
- テストが実際の変更に鈍感になる——動作が壊れてもパスし、動作が正常でも失敗する
- 先走ってしまい、実装を理解する前にテストの構造を固めてしまう

**正しいアプローチ**："tracer-bullet"を用いた垂直スライス。1つのテスト → 1つの実装 → これを繰り返す。各テストは、前のサイクルで学んだことに基づいて作成される。コードを書き終えたばかりなので、どの動作が重要で、それをどう検証すべきかが正確にわかっている。

```
WRONG (horizontal):
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

RIGHT (vertical):
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## Workflow

### 1. Planning

コードを書く前に：

- [ ] ユーザーと、どのようなインターフェースの変更が必要かを確認する
- [ ] ユーザーと、どの動作をテストすべきか（優先順位を付ける）を確認する
- [ ] [深層モジュール](deep-modules.md)（インターフェースは小さく、実装は深く）の適用機会を特定する
- [ ] [テスト可能性](interface-design.md)を考慮したインターフェースを設計する
- [ ] テストする動作をリストアップする（実装手順ではなく）
- [ ] 計画についてユーザーの承認を得る

「公開インターフェースはどのようなものにするべきか？どの動作をテストするのが最も重要か？」と尋ねる

**すべてをテストすることはできない。** どの動作が最も重要かをユーザーと正確に確認する。あらゆるエッジケースではなく、クリティカルパスや複雑なロジックにテストの労力を集中させる。

### 2. Tracer Bullet

システムに関する「1つのこと」を確認する「1つのテスト」を作成してください：

```
RED:   Write test for first behavior → test fails
GREEN: Write minimal code to pass → test passes
```

これがTracer Bulletです。つまり、経路が端から端まで正常に機能していることを証明します。

### 3. Incremental Loop

残りの各行動について：

```
RED:   Write next test → fails
GREEN: Minimal code to pass → passes
```

Rules:

- テストは1つずつ
- 現在のテストに合格するのに必要なコードのみ
- 将来のテストを先取りしない
- テストは観察可能な動作に焦点を絞る

### 4. Refactor

すべてのテストが成功したら、[リファクタリングの候補](refactoring.md)を探します：

- [ ] 重複の抽出
- [ ] モジュールの階層化（単純なインターフェースの背後に複雑さを隠す）
- [ ] 自然な箇所でSOLID原則を適用する
- [ ] 新しいコードが既存のコードについて何を明らかにしているかを検討する
- [ ] リファクタリングの各ステップ後にテストを実行する

**テストが「RED」の状態では絶対にリファクタリングを行わないでください。** まずは「GREEN」の状態にしてください。

## Checklist Per Cycle

```
[ ] テストは実装ではなく動作を記述している
[ ] テストはパブリックインターフェースのみを使用している
[ ] テストは内部のリファクタリング後も機能し続ける
[ ] このテストに必要なコードは最小限である
[ ] 仮定的な機能は追加されていない
```

## When to Activate

- 新機能や機能性の実装
- バグや問題の修正
- 既存コードのリファクタリング
- APIエンドポイントの追加
- 新しいコンポーネントの作成

## Best Practices

1. **テストを最初に書く** - 常にTDD
2. **テストごとに1つのアサーション** - 単一の動作に焦点を当てる
3. **説明的なテスト名** - 何をテストするか説明する
4. **Arrange-Act-Assert** - 明確なテスト構造
5. **外部依存をモック** - ユニットテストを分離
6. **エッジケースをテスト** - null、undefined、空、大きな値
7. **エラーパスをテスト** - ハッピーパスだけでなく
8. **テストを高速に保つ** - ユニットテストは各50ms未満
9. **テスト後にクリーンアップ** - 副作用を残さない
10. **カバレッジレポートをレビュー** - ギャップを特定

## Success Metrics

- 80%以上のコードカバレッジを達成
- すべてのテストが通過（グリーン）
- スキップまたは無効化されたテストがない
- 高速なテスト実行（ユニットテストは30秒未満）
- E2Eテストが重要なユーザーフローをカバー
- テストが本番前にバグを検出

---

**覚えておいてください**: テストはオプションではありません。テストは自信を持ったリファクタリング、迅速な開発、本番環境の信頼性を可能にするセーフティネットです。
