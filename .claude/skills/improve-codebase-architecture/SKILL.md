# コードベースのアーキテクチャ改善

AIのようにコードベースを分析し、アーキテクチャ上の課題を明らかにし、テスト容易性を向上させる機会を発見し、モジュールの深層化を目的としたリファクタリング案をGitHubのRFCとして提案する。

**ディープモジュール**（John Ousterhout著『A Philosophy of Software Design』）とは、大規模な実装を隠蔽する小さなインターフェースを持つモジュールのことです。ディープモジュールはテストしやすく、AIによる探索も容易であり、内部ではなく境界でテストを行うことを可能にします。

## Process

### 1. コードベースを探索する

Agentツールをsubagent_type=Exploreで実行し、自然な流れでコードベースを探索してください。厳格なヒューリスティックに従うのではなく、有機的に探索し、どこで違和感を覚えるかを記録してください：

- ある概念を理解するために、多くの小さなファイルを行き来しなければならない箇所はどこか？
- モジュールの階層が浅すぎて、インターフェースが実装とほぼ同じくらい複雑になっている箇所はどこか？
- テストしやすさだけを理由に純粋関数が抽出されているが、実際のバグはその呼び出し方法に潜んでいる箇所はどこか？
- 密結合なモジュールが、その接合部で統合リスクを生み出している箇所はどこか？
- コードベースのどの部分がテストされていない、あるいはテストしにくいのか？

遭遇する「摩擦」こそが、重要なシグナルなのです。

### 2. 候補の提示

深堀りの機会を番号付きリストとして提示してください。各候補について、以下を示してください：

- **クラスター**：どのモジュール／概念が関与しているか
- **結合の理由**：共有される型、呼び出しパターン、概念の共同所有
- **依存関係のカテゴリ**：4つのカテゴリについては [REFERENCE.md](REFERENCE.md) を参照
- **テストへの影響**：境界テストによって置き換えられる既存のテスト

現時点ではインターフェースを提案しないでください。ユーザーに「これらの中で、どの項目について検討したいですか？」と尋ねてください。

### 3. ユーザーが候補を選択する

### 4. 問題領域を定義する

サブエージェントを生成する前に、選択された候補について、ユーザー向けに問題領域の説明を作成してください：

- 新しいインターフェースが満たすべき制約
- 依存すべき要素
- 制約を具体的に示すための大まかなコードスケッチ — これは提案ではなく、単に制約を明確にするためのものです

これをユーザーに提示し、直ちにステップ5に進みます。サブエージェントが並行して動作している間、ユーザーはこれを読み、問題について検討します。

### 5. 複数のインターフェースを設計する

Agentツールを使用して、3つ以上のサブエージェントを並行して生成します。各エージェントは、深化させたモジュールに対して**根本的に異なる**インターフェースを生成する必要があります。

各サブエージェントには、個別の技術概要（ファイルパス、結合の詳細、依存関係のカテゴリ、隠蔽される内容）を提示する。この概要は、ステップ4のユーザー向けの説明とは独立している。各エージェントに異なる設計制約を与える：

- エージェント1：「インターフェースを最小限に抑える — エントリポイントは最大1〜3個を目標とする」
- エージェント2：「柔軟性を最大化する — 多くのユースケースと拡張をサポートする」
- エージェント3：「最も一般的な呼び出し元向けに最適化 — デフォルトケースを単純化せよ」
- エージェント4（該当する場合）：「境界を越える依存関係については、ポート＆アダプタパターンを中心に設計せよ」

各サブエージェントは以下を出力する：

1. インターフェースのシグネチャ（型、メソッド、パラメータ）
2. 呼び出し元がどのように使用するかを示す使用例
3. 内部で隠蔽している複雑さ
4. 依存関係戦略（依存関係の扱い方 — [REFERENCE.md](REFERENCE.md)を参照）
5. トレードオフ

設計案を順に提示し、その後文章で比較する。

比較後、自身の推奨案を提示する：どの設計が最も優れていると思うか、その理由を述べる。異なる設計の要素を組み合わせることで良い結果が得られる場合は、ハイブリッド案を提案する。明確な意見を示すこと — ユーザーは単なる選択肢の羅列ではなく、説得力のある解説を求めている。

### 6. ユーザーがインターフェースを選択する（または推奨案を受け入れる）

### 7. GitHubイシューを作成する

`gh issue create` コマンドを使用して、リファクタリングに関するRFCをGitHubイシューとして作成します。[REFERENCE.md](REFERENCE.md)にあるテンプレートを使用してください。作成前にユーザーにレビューを依頼しないでください。作成して、そのURLを共有するだけで十分です。
