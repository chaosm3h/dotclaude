# Reference

## 依存関係のカテゴリ

深層化の対象となる候補を評価する際は、その依存関係を次のように分類します：

### 1. プロセス内

純粋な計算、メモリ内の状態、I/Oなし。常に深層化可能 — モジュールをマージして直接テストするだけです。

### 2. ローカルで代替可能

ローカルなテスト用代替手段（例：Postgres の PGLite、インメモリファイルシステム）が存在する依存関係。テスト用代替手段が存在する場合、深層化可能。深層化されたモジュールは、テストスイート内でローカルな代替手段を実行してテストされる。

### 3. リモートだが所有されている（ポートとアダプタ）

ネットワーク境界を越えた自サービス（マイクロサービス、内部API）。モジュール境界でポート（インターフェース）を定義します。ディープモジュールがロジックを所有し、トランスポートは注入されます。テストではインメモリアダプタを使用します。本番環境では実際のHTTP/gRPC/キューアダプタを使用します。

推奨構成：「共有インターフェース（ポート）を定義し、本番環境用にはHTTPアダプターを、テスト用にはインメモリアダプターを実装します。これにより、ネットワーク境界を越えてデプロイされていても、ロジックを単一のディープモジュールとしてテストできます。」

### 4. 真の外部（モック）

制御できないサードパーティサービス（Stripe、Twilioなど）。境界でモック化します。深層化されたモジュールは外部依存関係を注入されたポートとして受け取り、テストではモック実装を提供します。

## テスト戦略

基本原則：**重ねるのではなく、置き換える。**

- 境界テストが存在するようになったら、浅いモジュールに対する古いユニットテストは無駄になるため、削除する
- 深くなったモジュールのインターフェース境界で新しいテストを書く
- テストでは、内部状態ではなく、パブリックインターフェースを通じて観測可能な結果に対してアサーションを行う
- テストは内部のリファクタリング後も機能し続けるべきである。テストは実装ではなく、動作を記述するものである

## 問題点

アーキテクチャ上の課題を説明してください：

- どのモジュールが浅く、密結合しているか
- それらの接点にどのような統合リスクが存在するか
- なぜこれがコードベースの理解や保守を困難にしているのか

## 提案するインターフェース

採用されたインターフェース設計：

- インターフェースのシグネチャ（型、メソッド、パラメータ）
- 呼び出し元がどのように使用するかを示す使用例
- 内部でどのような複雑さを隠蔽しているか

## 依存関係戦略

どのカテゴリが適用され、依存関係がどのように扱われるか：

- **インプロセス**：直接マージ
- **ローカル置換可能**：[特定の代用オブジェクト] を使用してテスト
- **ポートとアダプター**：ポート定義、本番用アダプター、テスト用アダプター
- **モック**：外部サービスに対するモック境界

## テスト戦略

- **新たに作成すべき境界テスト**：インターフェースで検証すべき動作を記述する
- **削除すべき既存のテスト**：不要になるモジュールの浅いテストをリストアップする
- **テスト環境の要件**：必要なローカルな代用オブジェクトやアダプター

## 実装に関する推奨事項

現在のファイルパスに依存しない、永続的なアーキテクチャ指針：

- モジュールが所有すべきもの（責任）
- 隠すべきもの（実装の詳細）
- 公開すべきもの（インターフェース仕様）
- 呼び出し元が新しいインターフェースへ移行する方法
